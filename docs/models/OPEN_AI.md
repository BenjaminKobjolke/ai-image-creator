# OpenAI Provider

How this project talks to OpenAI's Images API.

Code: `ai_image_creator/providers/openai_provider.py` · built by
`ai_image_creator/providers/registry.py`.

## Authentication

Key comes from `Settings` (`settings.py`), resolved in this order:

1. `--api-key` CLI flag (`Settings.load(api_key_override=...)`)
2. `OPENAI_API_KEY` environment variable
3. `OPENAI_API_KEY` in a `.env` file (loaded via `python-dotenv`)

`registry.build_provider("openai", settings)` raises a clear error if no key is
found. The key is never logged and never committed (`.env` is git-ignored).

## Models & sizes

Source of truth: `constants.VALID_SIZES` / `constants.DEFAULT_SIZE`. A request's
`size` is validated at the boundary (`models.ImageRequest`) and **rejected** if it
is not valid for the chosen model.

| Model                   | Valid sizes                             | Default size |
|-------------------------|-----------------------------------------|--------------|
| `gpt-image-2` (default) | `1024x1024`, `1024x1536`, `1536x1024`, `auto` | `1024x1024` |
| `gpt-image-1`           | `1024x1024`, `1024x1536`, `1536x1024`, `auto` | `1024x1024` |
| `dall-e-3`              | `1024x1024`, `1792x1024`, `1024x1792`   | `1024x1024` |
| `dall-e-2`              | `256x256`, `512x512`, `1024x1024`       | `1024x1024` |

`auto` lets the model pick. Default model is the newest (`gpt-image-2`); override
per request via JSON `model` or CLI `--model`.

> The installed `openai` SDK also exposes `gpt-image-1-mini`, `gpt-image-1.5`,
> `gpt-image-2-2026-04-21`, and `chatgpt-image-latest`. To use one, add it to
> `VALID_SIZES` + `DEFAULT_SIZE` in `constants.py` (confirm its real size list first).

## Request

`OpenAIProvider.generate(request)` calls `client.images.generate(...)` with:

| Arg      | From                     | Notes                                   |
|----------|--------------------------|-----------------------------------------|
| `model`  | `request.model`          |                                         |
| `prompt` | `request.prompt`         | required, non-empty                     |
| `size`   | `request.size`           | resolved/validated before the call      |
| `n`      | `request.n`              | images per request                      |
| `quality`| `request.quality`        | only sent when set (e.g. `high`)        |
| `background`| `request.background`  | only sent when set; `transparent` needs `gpt-image-1` |

The `OpenAI` client is **injected** into `OpenAIProvider` (constructor), so tests
pass a `MagicMock(spec=OpenAI)` — no network in unit tests.

### Reference images (edit path)

When a request carries `reference_images` (list of file paths), the provider routes
to `client.images.edit(model, image=[...], prompt, size, n, quality)` instead of
`generate`. The files are opened binary and closed via `contextlib.ExitStack`. Only
`gpt-image-2` / `gpt-image-1` are reference-capable (`constants.REFERENCE_CAPABLE_MODELS`);
the model layer rejects refs on `dall-e-*` before any call. The edit response uses
the same `b64_json` shape, so normalization is shared with `generate`. Both paths
build their args via `_base_kwargs`, so `quality` and `background` are forwarded to
edit too.

## Response normalization

OpenAI returns one of two shapes; the provider normalizes both to raw bytes
(`GeneratedImage.data`):

- **`gpt-image-*`** → base64 in `b64_json` → `base64.b64decode(...)`.
- **`dall-e-*`** → an image `url` → downloaded via `urllib.request.urlopen`.

If an item has neither, `generate` raises `ValueError`. If the API returns no data,
it returns an empty list.

## Output post-processing (provider-independent)

The provider only produces raw generated bytes at the provider `size`. Any final
resize/crop to the desired `output.width`/`output.height` happens afterward in
`processing/image_output.py` (Pillow) — see the two-size-concept note in the README.
This keeps the provider free of output concerns and identical across providers.

## Errors

Per-request failures (bad key, API error, unsupported combo) are caught by the batch
boundary in `runner.run_batch` and recorded as a failed `RequestResult`; one failure
does not abort the batch. Validation errors (invalid size/model) fail earlier, at
request construction, before any API call.

## Extending

To support another OpenAI model: add its id constant + valid sizes to `constants.py`.
To support another provider entirely: see "Adding a provider" in `CLAUDE.md`.
