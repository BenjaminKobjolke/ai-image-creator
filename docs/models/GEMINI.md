# Google Gemini Provider

How this project talks to Google's Gemini image APIs.

Code: `ai_image_creator/providers/gemini_provider.py` · built by
`ai_image_creator/providers/registry.py`.

## Authentication

Key comes from `Settings` (`settings.py`), resolved in this order:

1. `--api-key` CLI flag (`Settings.load(api_key_override=...)`)
2. `GEMINI_API_KEY` environment variable
3. `GEMINI_API_KEY` in a `.env` file (loaded via `python-dotenv`)

`registry.build_provider("gemini", settings)` raises a clear error if no key is
found. The `--api-key` flag feeds both OpenAI and Gemini, so one flag works for
whichever provider a request selects. The key is never logged and never committed
(`.env` is git-ignored).

## Models & sizes

Source of truth: `constants.VALID_SIZES` / `constants.DEFAULT_SIZE`. Gemini `size`
is an **aspect ratio** (not pixels). A request's `size` is validated at the
boundary (`models.ImageRequest`) and **rejected** if it is not valid for the model.

| Model                          | API path           | Valid sizes (aspect ratios)          | Default |
|--------------------------------|--------------------|--------------------------------------|---------|
| `gemini-2.5-flash-image`       | `generate_content` | `1:1`, `3:4`, `4:3`, `9:16`, `16:9`  | `1:1`   |
| `imagen-4.0-generate-001`      | `generate_images`  | `1:1`, `3:4`, `4:3`, `9:16`, `16:9`  | `1:1`   |
| `imagen-4.0-ultra-generate-001`| `generate_images`  | `1:1`, `3:4`, `4:3`, `9:16`, `16:9`  | `1:1`   |
| `imagen-4.0-fast-generate-001` | `generate_images`  | `1:1`, `3:4`, `4:3`, `9:16`, `16:9`  | `1:1`   |

> **Imagen models are deprecated by Google (shutdown 2026-08-17).** Prefer
> `gemini-2.5-flash-image`. They are included so existing JSON batches can still
> select them until then.

To add another Gemini/Imagen model: add its id constant + valid sizes to
`constants.py` (`VALID_SIZES` + `DEFAULT_SIZE`). `gemini-*` ids route to
`generate_content`; anything starting `imagen` routes to `generate_images`.

## Request

`GeminiProvider.generate(request)` routes by model id:

- **`imagen-*`** → `client.models.generate_images(model, prompt,
  config=GenerateImagesConfig(number_of_images=n, aspect_ratio=size))`.
- **`gemini-*`** → `client.models.generate_content(model, contents=prompt,
  config=GenerateContentConfig(response_modalities=["IMAGE"],
  image_config=ImageConfig(aspect_ratio=size)))`. This API returns one image per
  call, so it is looped `request.n` times.

`quality` is OpenAI-only and ignored here. The `genai.Client` is **injected** into
`GeminiProvider` (constructor), so tests pass a `MagicMock(spec=genai.Client)` — no
network in unit tests.

## Response normalization

Both shapes are normalized to raw bytes (`GeneratedImage.data`):

- **`imagen-*`** → `response.generated_images[i].image.image_bytes`.
- **`gemini-*`** → the first inline-image part:
  `response.candidates[*].content.parts[*].inline_data.data`.

If a response carries no image data, `generate` raises `ValueError`. If Imagen
returns no images, it returns an empty list.

## Output post-processing (provider-independent)

The provider only produces raw generated bytes at the requested aspect ratio. Any
final resize/crop to `output.width`/`output.height` happens afterward in
`processing/image_output.py` (Pillow) — identical across providers.

## Errors

Per-request failures (bad key, API error, unsupported combo) are caught by the
batch boundary in `runner.run_batch` and recorded as a failed `RequestResult`; one
failure does not abort the batch. Validation errors (invalid size/model) fail
earlier, at request construction, before any API call.
